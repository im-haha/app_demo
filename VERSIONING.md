# 版本发布规范（VERSIONING）

## 1. 目的

本文档用于统一本项目的版本编号、Tag 命名、发布规则与升级边界，避免出现以下问题：

- 不清楚什么时候应该从 `v1.0.x` 升级到 `v1.1.0`
- 不清楚什么时候应该从 `v1.x.x` 升级到 `v2.0.0`
- CI / GitHub Actions / Release Tag 命名不统一
- 团队成员对“重大版本”“功能版本”“补丁版本”的理解不一致

本规范适用于：

- Git Tag
- GitHub Releases
- Android APK 发布
- iOS 构建产物发布
- 对外版本说明与内部版本管理

---

## 2. 版本号格式

本项目采用 **Semantic Versioning（语义化版本）**：

```text
vMAJOR.MINOR.PATCH
```

例如：

- `v1.0.0`
- `v1.0.24`
- `v1.1.0`
- `v2.0.0`

各字段含义如下：

| 字段 | 含义 | 说明 |
|---|---|---|
| `MAJOR` | 主版本号 | 存在不兼容变更（Breaking Change）时递增 |
| `MINOR` | 次版本号 | 新增功能且保持向后兼容时递增 |
| `PATCH` | 修订号 | Bug 修复、配置修正、构建修复、性能微调等向后兼容修改时递增 |

---

## 3. 版本递增规则

### 3.1 PATCH：补丁版本

以下情况发布 `PATCH` 版本：

- 修复线上 bug
- 修复 UI 显示问题
- 修复 Android / iOS 构建问题
- 修复 CI / Release Workflow 问题
- 修正日志、监控、Sentry、配置等非破坏性问题
- 不改变现有用户使用方式的代码优化

示例：

- `v1.0.24 -> v1.0.25`
- `v1.1.3 -> v1.1.4`

### 3.2 MINOR：功能版本

以下情况发布 `MINOR` 版本：

- 新增明确的功能模块
- 增加新页面、新能力、新交互流程
- 新增但不破坏旧接口的 API
- 发布能力明显增强，但旧版本调用方式、配置方式、升级方式仍兼容

示例：

- `v1.0.24 -> v1.1.0`
- `v1.1.4 -> v1.2.0`

**注意：** 发布新的 `MINOR` 版本时，`PATCH` 必须归零。

### 3.3 MAJOR：主版本

以下情况发布 `MAJOR` 版本：

- 存在 **不兼容变更（Breaking Change）**
- 老版本客户端无法平滑升级
- API 返回结构或调用方式发生破坏性变化
- 核心配置方式发生变化，旧配置不可继续使用
- 包体形态、平台策略、升级链路发生重大变化
- 数据结构、缓存结构、持久化格式发生不兼容变更
- 用户的核心操作路径发生重大调整，旧文档/旧操作方式不再适用

示例：

- `v1.9.8 -> v2.0.0`
- `v2.4.3 -> v3.0.0`

**注意：** 发布新的 `MAJOR` 版本时，`MINOR` 和 `PATCH` 都必须归零。

## 4. `v1.0.0` 和 `v2.0.0` 的边界定义

本项目对 `v2.0.0` 的定义不是“版本号涨到一定程度自动升级”，而是：

> 只有当本次发布包含明确的破坏性变更或重大里程碑变化时，才允许从 `v1.x.x` 升级到 `v2.0.0`。

### 4.1 可以判定为 `v2.0.0` 的情况

满足以下任意一项，可评估升级为 `v2.0.0`：

1. **接口不兼容**
   - 旧客户端请求新服务端失败
   - 响应字段发生删除、重命名、结构重组
   - 旧调用方式失效

2. **配置不兼容**
   - 旧 `.env` 或构建参数无法继续使用
   - 新版强制依赖新的签名文件、环境变量、发布参数
   - 旧 CI 配置无法直接复用

3. **安装 / 升级链路变化明显**
   - 包体从单包切换为多包，且旧升级方式不兼容
   - 渠道、架构、设备类型的发布策略发生重大变化
   - 安装后的行为与旧版本体系不同

4. **数据结构不兼容**
   - 本地缓存格式变更导致旧数据不可用
   - 数据库 schema 变更且无法平滑迁移
   - 历史持久化数据需要一次性迁移甚至清空

5. **核心产品形态升级**
   - 业务流程重构
   - 主要用户路径变化
   - 旧版使用说明书、部署说明、接口文档不再适用

### 4.2 不应升级为 `v2.0.0` 的情况

以下情况通常 **不应** 升级为 `v2.0.0`：

- 只是修复 bug
- 只是优化 UI
- 只是提升性能
- 只是补充监控 / 埋点 / Sentry
- 只是增加兼容性功能
- 只是新增页面，但旧流程仍可继续使用
- 只是 CI 工作流修正
- 只是 Android / iOS 构建脚本修复

这类情况应优先考虑：

- `PATCH`：小修复
- `MINOR`：新功能但兼容旧逻辑

## 5. 本项目推荐的版本使用策略

结合当前项目实际情况，推荐采用以下规则：

### 5.1 常规修复发布

使用：

```text
v1.0.x
```

适用场景：

- 修复 release workflow
- 修复 Android 打包问题
- 修复 iOS release artifact 问题
- 修复白屏闪退、配置缺失、签名缺失、Sentry 映射等问题

### 5.2 新功能发布

使用：

```text
v1.1.0
v1.2.0
v1.3.0
```

适用场景：

- 新增业务模块
- 增加新的发布能力
- 支持新的设备形态
- 增加新的用户功能，但保持原逻辑兼容

### 5.3 重大升级发布

使用：

```text
v2.0.0
```

适用场景：

- 存在明确 breaking change
- 对外需要宣布“新一代版本”
- 技术架构或产品交互发生显著改变
- 旧客户端、旧配置、旧流程不再兼容

## 6. Tag 命名规范

本项目发布 Tag 统一采用以下格式：

```text
vMAJOR.MINOR.PATCH
```

正确示例：

- `v1.0.24`
- `v1.1.0`
- `v2.0.0`

错误示例：

- `1.0.24`  ← 缺少前缀 `v`
- `release-1.0.24`  ← 不统一
- `V1.0.24`  ← 大小写不统一
- `v1.0`  ← 不完整

### 6.1 原则

- Git Tag、GitHub Release、CI 触发版本号必须保持一致
- 不允许同一版本号重复打不同内容的 Tag
- 已公开使用的版本 Tag 不应强行重写

---

## 7. 分支与发布建议

### 7.1 推荐发布来源

正式版本建议从以下分支之一创建 Tag：

- `master`
- `main`

应避免从以下分支直接发布正式版本：

- `feature/*`
- `fix/*`
- `test/*`
- 临时调试分支

### 7.2 发布前要求

在创建 Release Tag 之前，应至少满足：

- 目标分支代码已合并完成
- CI Quality Gate 通过
- 对应平台构建通过
- 核心功能已完成验证
- 必要配置、签名、环境变量齐全

---

## 8. 发布决策流程

发布前按以下顺序判断版本号：

### Step 1：是否存在不兼容变更？

- 是 → 升级 `MAJOR`
- 否 → 进入 Step 2

### Step 2：是否新增了向后兼容的新功能？

- 是 → 升级 `MINOR`
- 否 → 进入 Step 3

### Step 3：是否只是修复、优化、配置调整？

- 是 → 升级 `PATCH`

---

## 9. 发布示例

### 示例 1：修复 Android release 闪退

情况：

- 修复缺失配置
- 修复签名或构建问题
- 不影响现有 API 和用户操作方式

版本建议：

```text
v1.0.24 -> v1.0.25
```

### 示例 2：新增 Pad 与 Phone 独立 APK 发布能力

情况：

- 新增构建能力
- 旧使用方式仍兼容
- 不破坏已有客户端升级链路

版本建议：

```text
v1.0.24 -> v1.1.0
```

### 示例 3：接口全面改版，旧客户端不兼容

情况：

- 服务端返回结构变化
- 旧前端请求逻辑失效
- 需要用户整体升级

版本建议：

```text
v1.3.7 -> v2.0.0
```

### 示例 4：构建系统、配置系统和发布流程全部重构

情况：

- 旧 CI 配置不可继续用
- 旧发布文档全部失效
- 旧包管理方式不再适用

版本建议：

```text
v1.x.x -> v2.0.0
```

## 10. 预发布版本（可选）

如果后续需要区分测试版、候选发布版，可采用以下扩展格式：

```text
v1.2.0-beta.1
v1.2.0-rc.1
```

定义：

- `beta`：测试版，功能基本完成但仍需验证
- `rc`：Release Candidate，候选正式版，原则上无阻塞问题即可转正式版

当前若团队尚未形成稳定预发布流程，可暂不启用。

---

## 11. 版本发布记录要求

每次正式发布建议至少记录以下内容：

- 版本号
- 发布时间
- 发布人
- 对应 Git Tag
- 主要变更内容
- 是否包含 breaking change
- 是否需要升级说明 / 数据迁移说明 / 回滚说明

建议在 GitHub Release 中维护简要 Release Notes。

---

## 12. 团队执行约定

为避免版本滥用，统一执行以下约定：

1. 不允许为了“看起来版本更大”而随意提升 `MAJOR`
2. 不允许把普通 bug 修复发布成 `MINOR`
3. 不允许把新增功能继续堆在 `PATCH` 中发布
4. 对是否属于 breaking change 存在争议时，以“旧版本是否仍可无痛使用”为最高判断标准
5. 正式发布前，版本号应在 PR、Tag、Release Notes 中保持一致

---

## 13. 一句话规则

可直接记为：

> 修 bug 用 `PATCH`，加功能用 `MINOR`，有破坏性变更才升 `MAJOR`。

即：

- 修复问题：`v1.0.x`
- 兼容性新增功能：`v1.x.0`
- 不兼容或重大升级：`v2.0.0`

---

## 14. 当前项目建议

基于当前项目发布现状，建议默认采用如下策略：

- 当前小版本阶段持续使用：`v1.0.x`
- 当新增明确功能并保持兼容时：升级到 `v1.1.0`、`v1.2.0`
- 当出现明确 breaking change 或项目形态升级时：升级到 `v2.0.0`

换句话说：

**`v2.0.0` 不是数字自然增长的结果，而是一次有明确边界定义的主版本升级。**
